# 多部门用户体系设计方案

> 日期：2026-03-09
> 状态：待实现

## 一、需求背景

当前系统采用「用户-项目」两级结构，所有用户共享同一个空间。当多个团队（部门）使用同一系统时，需要实现数据隔离和权限控制。

### 需求确认

| 维度 | 选择 |
|------|------|
| 使用场景 | 企业内部多部门 |
| 用户归属 | 单部门归属（每人只属于一个部门） |
| 跨部门协作 | 项目级邀请（被邀请者只能访问特定项目） |
| 管理层级 | 两层（系统管理员 + 部门管理员） |

---

## 二、核心概念

### 部门（Department）

- 类似公司的部门，如研发部、市场部、财务部
- 每个部门名称唯一
- 每个部门指定一个管理员

### 三种角色

| 角色 | 谁来担任 | 能做什么 |
|------|----------|----------|
| **系统管理员** | 公司IT或老板 | 创建/删除部门、指定部门管理员、管理所有数据 |
| **部门管理员** | 部门负责人 | 管理本部门成员、创建/管理本部门项目 |
| **普通成员** | 员工 | 只能操作自己参与的项目 |

### 人员归属规则

1. **每个员工只属于一个部门**
2. **每个部门只有一个管理员**
3. **系统管理员可以不属于任何部门，但能查看和管理所有部门**

### 项目归属规则

1. **项目属于部门**：创建项目时自动继承创建者的部门
2. **部门成员可见本部门所有项目**
3. **跨部门协作通过邀请**：被邀请者只能访问特定项目

## 三、数据模型设计

### 新增 Department 表

```prisma
// 部门表
model Department {
  id          String   @id @default(uuid())
  name        String   @unique  // 部门名称唯一
  description String?
  adminId     String             // 部门管理员（一对一）
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  // 关联
  admin    User        @relation("DepartmentAdmin", fields: [adminId], references: [id])
  members  User[]      @relation("DepartmentMember")
  projects Project[]
}
```

### 修改 User 表

```prisma
model User {
  // ... 现有字段保持不变

  // 新增字段
  departmentId String?  // 所属部门（可为空，兼容系统管理员）

  // 新增关联
  department        Department?  @relation("DepartmentMember", fields: [departmentId], references: [id])
  managedDepartment Department?  @relation("DepartmentAdmin")  // 管理的部门（如果有）
}
```

### 修改 Project 表

```prisma
model Project {
  // ... 现有字段保持不变

  // 新增字段
  departmentId String?  // 所属部门（可为空，兼容迁移期）

  // 新增关联
  department Department? @relation(fields: [departmentId], references: [id])
}
```

## 四、权限规则

### 系统管理员权限

| 操作 | 权限 |
|------|------|
| 创建/删除部门 | ✅ |
| 指定/更换部门管理员 | ✅ |
| 查看所有部门数据 | ✅ |
| 查看所有项目 | ✅ |
| 添加/移除部门成员 | ✅ |
| 重置用户密码 | ✅ |

### 部门管理员权限

| 操作 | 权限 |
|------|------|
| 添加本部门成员 | ✅ |
| 移除本部门成员 | ✅ |
| 创建本部门项目 | ✅ |
| 管理本部门所有项目 | ✅ |
| 邀请其他部门人员参与项目 | ✅ |
| 查看其他部门 | ❌ 只能看到成员列表（用于邀请） |
| 创建新部门 | ❌ |

### 普通成员权限

| 操作 | 权限 |
|------|------|
| 查看本部门项目 | ✅ |
| 参与的项目中创建/编辑任务 | ✅ |
| 邀请本部门成员参与项目 | ✅（需项目管理员批准） |
| 邀请其他部门成员 | ❌ |
| 查看其他部门项目 | ❌ |

### 跨部门邀请规则

**流程**：
1. 项目负责人在项目成员管理中搜索其他部门成员
2. 系统显示该成员的基本信息（仅姓名、部门）
3. 点击邀请
4. 被邀请者收到通知，选择接受或拒绝
5. 接受后，成为项目协作者
6. 权限仅限该项目，不包括邀请方部门的其他项目

**限制**：
- 只有项目负责人和部门管理员可以发起跨部门邀请
- 被邀请者在项目中的角色为「协作者」，不能成为项目负责人

## 五、界面变更

### 新增页面

**部门管理页**（系统管理员专属）
- 部门列表：显示所有部门、成员数、管理员
- 创建部门：输入名称、描述、指定管理员
- 编辑/删除部门

**我的部门页**（部门管理员专属）
- 本部门成员列表
- 添加/移除成员
- 查看本部门项目统计

### 修改现有页面

**注册页面**
- 新增：选择所属部门（下拉选择）
- 系统管理员审核后才能加入部门

**项目管理页**
- 筛选：只显示本部门项目
- 新增「部门」标签显示

**项目详情页**
- 成员管理：增加跨部门邀请功能
- 显示成员所属部门

**任务表单**
- 负责人选择：默认只显示本部门成员
- 增加搜索其他部门成员的能力（仅显示姓名和部门）

**导航栏**
- 显示当前用户所属部门
- 部门管理员增加「部门管理」入口

### 用户信息展示

凡是显示用户名的地方，增加部门标识：
- 任务列表：负责人后面显示「[研发部]」
- 项目成员列表：每个成员显示所属部门
- 评论、通知等：显示用户部门

## 六、数据迁移方案

### 迁移步骤

**第一步：创建部门**
- 系统管理员创建各部门
- 指定各部门管理员

**第二步：分配现有用户**
- 将现有用户分配到对应部门
- 未分配的用户暂时放入「未分配」状态

**第三步：迁移现有项目**
- 根据项目创建者的部门，自动归属
- 创建者未分配部门的，项目暂时无部门归属

**第四步：清理**
- 处理跨部门项目的特殊情况
- 验证数据完整性

### 兼容性处理

| 情况 | 处理方式 |
|------|----------|
| 用户未分配部门 | 可以登录，但提示联系管理员分配部门 |
| 项目无部门归属 | 仍可正常访问，但提示管理员设置部门 |
| 现有邀请记录 | 保持有效，不受部门限制 |

### 回滚方案

- 数据库保留迁移前的备份
- 部门相关字段均为可空，不影响原有功能
- 可以逐步启用部门功能，而非一次性切换

## 七、示例场景

```
公司结构：
├── 系统管理员：老赵（IT总监）
│
├── 研发部（管理员：王经理）
│   ├── 成员：张三、李四、小王
│   └── 项目：APP开发、后台重构
│
└── 市场部（管理员：刘总监）
    ├── 成员：小红、小明
    └── 项目：双11活动、品牌推广
```

**日常场景**：
- 张三创建「APP开发」项目 → 自动归属研发部
- 研发部所有人都能看到这个项目
- 张三邀请市场部小红参与 → 小红能看到这个项目
- 但小红看不到研发部的「后台重构」项目
